Smarter-Barter (Time-Bank) - Springboard UX Design School Project

This is the final project for Udacity's course Intro to the Design of Everyday Things taught by Don Norman, Kristian Simsarian and Chelsey Glasson. My first few iterations of this project were part of the Springboard UX Design School.

The project consisted of designing a user interface for a time-bank that allows people to exchange their time for another person's time. Everyone's time has equal value. Some of the proposed functionality for the project included: finding service providers, selecting a service provider, agreeing upon the transaction and indicating the balance once a transaction has been completed.

Login and Registration

Find and Schedule a Service

Landing page for the app. Users can immediately see content and become engaged. Requires registration/login before scheduling a service.

Users can search by keyword or name to find a service or service provider. Tapping the calendar will initiate scheduling. Tapping the provider's name will show a detailed profile with reviews.

Only available dates will appear for scheduling. User can cancel scheduling at any sep in the process.

Only available times will appear for scheduling.

User is shown the service, service provider, date/time and the cost/balance before confirmation.

The user will be redirected to the schedule screen where the service will appear as pending until the service provider accepts the request for service.

When the service provider accepts the service, the requesting user will receive an alert that the service has been confirmed.

The requesting user will recieve an alert if the service provider passes on the service request.

Schedule Views

The schedule menu initially provides a listing of all services that you are to recieve and give. The user can also filter by receive or give.

Any listing can be tapped on to show details of that service. The user can swipe through to the next service within the detail view.

When you provide a service, the detail view will display a map of the requesting user's location.

Alerts and associated Action Tasks

The Alerts screen is for asynchronous notifications (requests, confirmations, reschedules, etc.) for which the user may need to take action upon or updated on pending statuses.

When the user taps on an alert, they can receive details that may require action. In this case it's a request for a repair that will require the user to accept or pass on the request.

If the user accepts the request, the screen will redirect to the schedule listing showing the service added to the schedule.

If the user passes on the service request, they will be redirected automtically to the alerts screen to take action on any remaining alerts.

Feedback

Feedback can be viewed for each user on the profile detail screen. The user can reach this screen by browsing for services or doing a direct search under the find menu.

After services have been provided, the user that received the service will receive an alert to leave feedback for the service provider. Feedback is optional and the user can swipe right to remove the notification.

The user is provided with the service provider, service provided and date on the feedback form to provide readily available context for the feedback.
